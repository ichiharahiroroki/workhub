# AI Agent Operating Rules

このファイルは、このディレクトリで作業するAIエージェントの振る舞いを定義するものです。
アプリ仕様・UI仕様・機能仕様はここに書かない。仕様や確認事項は `prototype-communication.md` に書く。

## 役割

- 親エージェントは、ユーザーの意図を確認し、作業方針を整理し、必要な実装を統合する。
- サブエージェントは、親エージェントが明示した独立タスクだけを担当する。
- どのエージェントも、ユーザーの意図を推測だけで確定しない。

## 最優先ルール

- `beyond-r-delivery-manage.md` は原典として扱う。変更しない。
- 直近のユーザー指示を最優先する。
- 既存実装より、ユーザーの説明・原典・認識合わせメモを優先する。
- 仕様判断に迷う場合は、実装でごまかさず `prototype-communication.md` に確認事項を書く。
- 見た目だけで「それっぽい」ものを作らない。
- ユーザーが求めているものが不明確な場合、先に認識合わせをする。

## 参照順序

1. 直近のユーザー指示
2. `prototype-communication.md` の確認事項・合意事項
3. `beyond-r-delivery-manage.md`
4. ユーザーが共有した参考ページ・画像・説明
5. 既存の実装

既存実装が上記と矛盾する場合、既存実装を正としない。

## 作業前の行動

親エージェントは、大きな実装や修正に入る前に以下を行う。

- ユーザーの指摘を要約する。
- 何を直すのかを `prototype-communication.md` に記録する。
- 原典のどの記述と関係するかを確認する。
- 不明点があれば `prototype-communication.md` の `確認したいこと` に書く。
- 実装してよい部分と、認識合わせが必要な部分を分ける。

## コミュニケーション

- ユーザーとは日本語で会話する。
- 判断に迷ったら、まず `prototype-communication.md` に書く。
- 質問は具体的にする。
- 「よさそう」「それっぽい」など曖昧な言葉で仕様を確定しない。
- ユーザーの不満や指摘がある場合、反論より先に原因を整理する。

## サブエージェント利用

- サブエージェントは、依存関係のない量産タスクにのみ使う。
- サブエージェントに画面全体の設計判断を任せない。
- サブエージェントには、担当ファイル、変更禁止ファイル、期待する成果物を明記する。
- サブエージェントには、`beyond-r-delivery-manage.md` を変更させない。
- サブエージェントの成果物は、親エージェントが必ず確認してから統合する。

## 禁止事項

- `beyond-r-delivery-manage.md` を編集すること。
- アプリ仕様を `agents.md` に書くこと。
- ユーザー確認が必要な仕様を、勝手に実装で確定すること。
- 指示されていない外部ページ遷移、外部検索、外部サービスへのリンク遷移を実装すること。
- 仕様理解が不十分なまま、UI部品を増やして進捗に見せること。
- 「カードを並べるだけ」「表を置くだけ」で、ユーザーが求める操作体験を満たしたことにすること。

## 実装後の行動

実装した場合、親エージェントは以下を行う。

- 変更ファイルを明示する。
- 何を直したかを、ユーザーの指摘に対応づけて説明する。
- ビルドや表示確認を行う。
- 原典ファイルを変更していないことを確認する。
- 残っている確認事項を `prototype-communication.md` に残す。
