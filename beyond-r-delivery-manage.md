# Beyond R デリバリーマネジメント・プレイブック

## このドキュメントについて

このドキュメントは、Beyond R というリソースマネジメントツールの土台となる考え方・構造・定義をまとめたものである。

Beyond R は、少人数チームが、限られた時間と予算の中で、現実的に前へ進むための判断を支えるツールを目指す。
そのために、Calendar との連携しながら、リソースの器・配分・残量・余白を扱う。

本ドキュメントでは以下を定義する：

1. Philosophy — なぜこのツールが必要か、何を解決するか
2. Core Concepts — このツールを理解するための核となる考え方
3. Definitions — 用語と数値の定義
4. Structure — 管理区分と構造
5. Operations — 具体的な運用ルール
6. Open Questions — 未決事項と将来の拡張

---

## 1. Philosophy（なぜこのツールが必要か）

### 1.1 Beyond R が解決すること

プロジェクト型のアサインもあれば、単発で依頼されるタスクもある。学習や探究、実験や挑戦の要素が強いタスクもある。
多方向にコミットメント（アサイン）する働き方をしている人にとって、以下のようなことが起きやすい：

- 複数のプロジェクトと単発タスクが同時に走り、全体像が見えなくなる
- 突発の依頼が入ることで、計画していた作業が押し出される
- 学習や実験のような将来への投資に、定量的な工数管理ができていない
- 多方向からタスクが飛んできても、リソースやタスクの追跡ができない

Beyond R は、こうした状況において **リソースの全体像を可視化し、現実的な判断を支える** ためのツールである。

### 1.2 管理の起点は「タスク一覧」ではなく「リソース」

Beyond R では、管理の起点をタスク一覧やチケット一覧ではなく、
**使えるリソースの総量** に置く。

中心にある問い：

- 今のチームに、どれだけの時間が使えるのか
- その時間の中に、何をどれだけ入れられるのか
- 何を入れると、何が溢れるのか
- どこに時間が使われているのか

つまり Beyond R は、
「やることを全部管理する」ためのツールではなく、
**「限られたリソースの中で、現実的な判断をする」ためのツール** である。

### 1.3 溢れないように配分するのが前提

Beyond R は「無理が発生したら警告する」ツールではなく、
そもそも無理のないタスク配分をするためのツールである。

- 処理可能な工数を超えるタスクが来たら、優先順位を判断し、優先度の低いものを後ろに回す
- 「溢れてから対処する」のではなく「溢れないように配分する」

### 1.4 ゴールとリソースを同時に見る

Beyond R は2つの視点を持つ。

| 画面 | 役割 | 答える問い |
|---|---|---|
| リソース画面 | リソースと配分を見る | 今どれだけ余裕があるか？どこに使われているか？ |
| ドリルダウン画面 | タスクを階層的・分類的に見る | 各ゴールに対して今どこまで進んでいるか？ |

この2つはデータを共有する。タスクに紐づく工数実績が両方の画面に流れる。
すべてのタスクは必ずドリルダウン画面上のいずれかの分��に属する。リソース画面にはあるがドリルダウン画面にはない、という状態は存在しない。

### 1.5 AI と人間の協調

Beyond R は、人間だけでなく AI との協調も前提としたコラボレーションツールを目指す。
タスクを Slack、Notion、GitHub などの外部サービスと紐づけることで、AI と人間の双方が同じ情報を共有し、タスクを追いやすくする。

- タスクと成果物に Slack 作業スレ、Notion ページ、Drive のリンクを紐づける
- AI が文脈を理解した状態で支援や提案を行えるようにする
- AI に実際にタスクを委任し、成果物を納品できるレベルまで将来的に開発する
- AI の権限設定や作業委任の管理も、このツール上で調整できることを理想とする

このプロトタイプはその方向性を示すものであり、将来的な実装のイメージを共有するためのものでもある。

## 2. Core Concepts（核となる考え方）

### 2.1 時間（hour）ベースで管理する

Beyond R はリソースマネジメントツールであり、Calendar との突き合わせが本質的に重要である。
そのため、工数は **時間（hour）ベース** で管理する。

- 各タスクは必ず hour 単位で見積もる
- 見積もりコストを下げるために、見積もり算出用の AI を用意する
- 実績から正しい戦闘力を計算し、見積もり精度を継続的に改善する

### 2.2 Beyond R でタスク管理を一元管理する

Beyond R は、すべてのタスク（Project / Ad-hoc / Explore）を一元で管理する。

タスクの定義、作成、更新、完了は Beyond R 内で行われ、すべてのステータス・工数・進捗が Beyond R に記録される。

- タスクの作成・編集・削除は Beyond R で行う
- 関連する成果物（Slack スレ、Notion ページ、Drive ファイルなど）はリンク・参照として紐づける
- Beyond R がタスク情報の唯一の情報源となり、複数ツール間での齟齬がなくなる

### 2.3 Calendar からリソースを算出する

Beyond R は Google Calendar（主に想定）と連携し、日ごとの現実的に使える時間を抽出する。

カレンダー上の予定は Reserved として取り込まれ、それを差し引くことで正確な実稼働時間を算出する。
全体会議や朝会など、具体的にどの成果物に結びつけるか分からないようなリソース消費を可視化するために重要な仕組みである。

### 2.4 戦闘力で個人差を定量化する

複数人で作業するとき、誰がどれだけのタスクを捌くことができるかを定量的に数値化する必要がある。

そのために、基準となる「4年目くらいの一般的なエンジニア」を軸（= 1.0）として、各作業者の戦闘力を数値化する。

- タスクの見積もりは戦闘力 1.0 を基準として算出する
- 各作業者の戦闘力 × 実稼働時間 = その人が処理できる工数
- 実績が溜まることで、過去の実績や得意分野から AI が戦闘力を算出・更新する

### 2.5 見積もりが難しいものへの対処

- 過去の経験（似たケースか、初めての挑戦か）によって工数計算を調整する
- キャッチアップ用のタスクを別途切ることで、学習工数を明示的に確保する
- 見積もりが難しいタスクは、まず「AI に相談して段取りを決めるタスク」を切る
- 作ったことのないシステムの設計などは、Explore でチケットを切る
- 見積もり算出用のドキュメントを整備し、AI と合わせて活用する

### 2.6 管理区分で仕事の性質を分ける

Beyond R では、仕事を Project / Ad-hoc / Explore の3つの管理区分で扱う。
これとは別に、カレンダー上で時間を占有する予定を Reserved として扱う。

- Project — ゴールがあり、まとまりがあり、スプリントで回す対象
- Ad-hoc — 単発の依頼や突発対応。カテゴリー分け可能
- Explore — 学習・探究・実験・挑戦。定量的に工数管理する
- Reserved — カレンダー上の拘束時間。実稼働時間の計算に使う

詳細は「4. Structure」で定義する。

### 2.7 定量的な実績を可視化し報告する

Beyond R は、個人やチームの実績を定量的に算出し、定期的にレポートとして提出できるようにする。

- その期間に達成したことを伝える
- 実戦闘力・完了タスクの見積もり工数合計などを定量評価として提示する
- 役員会への報告に使える形式で出力する

## 3. Definitions（用語と数値の定義）

### 3.1 予想戦闘力

対象の人が実際に作業に使える能力を表したもの。

```text
予想戦闘力 =（営業日 - 非作業日 - Reserved）× 8 × アサイン比率 × 実タスク比率 × 潜在能力
```

| 要素 | 説明 |
|---|---|
| 営業日 | 対象期間の営業日数 |
| 非作業日 | 有給・祝日など |
| Reserved | Calendar 上���拘束予定（会議、定例、移動等） |
| 8 | 1日の基準労働時間（hour） |
| アサイン比率 | そのプロジェクト/チームへの配分割合 |
| 実タスク比率 | 稼働時間のうち、実際にタスクに使える割合 |
| 潜在能力 | 4年目エンジニア = 1.0 を基準とした相対能力 |

### 3.2 投下工数

タスクの投下工数の合計。実際に成果物の作成にかけた時間。
各作業者の実戦闘力を計算する土台となる。

### 3.3 実戦闘力

投下工数に潜在能力をかけたもの。

- 予想戦闘力と比較しての大小で色付けするが、単純に大きければよく小さければ悪いものではない
- 予想戦闘力と実戦闘力が乖離している理由を振り返り、スプリントごとに実タスク比率・潜在能力・タスクの見積もり時間を見直す
- 完了タスクはそのまま実績として計算する
- レポート作成時点で未完了のタスクについては、完了率を入力して算出する

## 4. Structure（管理区分と構造）

### 4.1 管理区分の定義

#### Project

| 項目 | 内容 |
|---|---|
| 定義 | ゴールがあり、まとまりがあり、複数の成果物やタスクがつながっている仕事。スプリントで計画・実行・振り返りを回す対象であり、Beyond R でタスク一式を管理する |
| 例 | 社内システムのアジャイル開発, 業務自動化推進PJ |

#### Ad-hoc

| 項目 | 内容 |
|---|---|
| 定義 | Project に属さない単品の仕事。依頼や発生事象を起点に発生し、カテゴリー分けして管理する。時間と紐づく定期業務は Calendar（Reserved）で扱う |
| 例 | 単発の分析依頼, 問い合わせ対応, 軽微な調査, 細かな改善 |

#### Explore

| 項目 | 内容 |
|---|---|
| 定義 | 自発的に取り組む学習・探究・実験・挑戦。工数管理がしづらく定量的に管理しにくい探究活動を、あえて管理区分として独立させ、見積もりと実績を継続的に改善する |
| 例 | 新技術の学習, SaaS/API 検証, AI活用実験, 自動化の試作 |

#### Reserved

| 項目 | 内容 |
|---|---|
| 定義 | 管理区分ではない。カレンダー上で時間を占有する予定。タスク管理の対象にはせず、実稼働時間の計算に使う |
| 例 | 全社会議, 朝会, 休暇, 移動, 定期業務（時間が固定のもの） |

### 4.2 基本構造

Project の構造：

```text
Project
  └ マイルストーン
      └ 成果物
          └ タスク
```

Ad-hoc / Explore / Reserved の構造：

```text
管理区分（Ad-hoc / Explore / Reserved）
  └ カテゴリ（種類）
      └ タスク
```

- Beyond R 内で、すべてのタスク定義・更新・完了を管理する
- 詳細なタスクや課題は、別途の成果物管理ツール（Notion など）で補足されることもあるが、Beyond R が工数・進捗の唯一の情報源となる
- Beyond R に紐づかないタスクは存在しない

## 5. Operations（運用ルール）

### 5.1 スプリントの運用

- スプリントは主に Project に適用する
- 一定期間で何を進めるかを定め、その範囲が現実的かを確認する
- 実績との差分を見て次回に活かす
- Ad-hoc / Explore はスプリントで厳密にコミットしなくてもよいが、スプリント区切りで消費量を振り返る
- スプリントごとに実タスク比率・潜在能力・タスクの見積もり時間を見直す

### 5.2 タスクへの外部リンク紐づけ

実際に行ったタスクには、以下を必要に応じて紐づけられるようにする：

- AI セッション（チャットログ）
- Slack の作業スレ（タスクごとにスレッドを立て、メモ・判断軸・思いつきを記録）
- Notion ページ
- Google Drive のファイルパス

ただし、タスクの定義・進捗・工数はすべて Beyond R に記録され、Beyond R が情報源となる。

### 5.3 マイルストーンの現実性判断

Beyond R は、マイルストーンを現実的にするための道具である。

- 関連タスクや作業総量が見える
- その期間の実稼働時間が見える
- Ad-hoc や Explore の消費も含めて判断できる
- 過去実績をもとに見積もりを改善できる

Beyond R が支えたい判断：

- このマイルストーンは今のリソースで達成可能か
- 無理なら、何を削るべきか / 後ろ倒しにすべきか
- Ad-hoc を受ける余地があるか

### 5.4 データ鮮度の維持

- Project は定例会議でリソースとやるべきことの認識を合わせ、そこでチケットを切る
- Ad-hoc タスクはその都度、依頼を受ける度に Beyond R に追加する
- 投下工数は日々、またはスプリント単位で入力する
