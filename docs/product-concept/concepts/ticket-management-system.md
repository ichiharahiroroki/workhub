# チケット管理制

このドキュメントでは、WorkHub におけるチケット管理の仕組みを説明する。

WorkHub では、チケットを仕事の文脈を集約する単位として扱う。何をするかだけでなく、目的、背景、関連情報、レビュー観点、見積工数、投下工数などをチケット上で扱い、人間とAIが同じ文脈を参照できるようにする。

一般的に言われるチケット制との違いは、[一般的なチケット制との比較](../references/ticket-management-comparison.md) にまとめる。

## WorkHub における役割

- Project / Ad-hoc / Explore など性質の異なる仕事を、[管理区分](./work-management-categories.md) に応じて整理する
- GitHub Issue / PR、Notion ページ、Google Drive ファイル、Slack スレッド、AI セッションなどを仕事単位で紐付ける
- 人間とAIが同じ文脈を参照できる状態をつくる
- 見積工数、投下工数、進捗、期限をもとに実行可能性を判断する
- 完了後も判断履歴や成果物を追跡できるようにする

## チケットの構造

### Project

Project のチケットは、マイルストーン、グループ、成果物、タスクの階層で管理する。

```text
Project
  └ マイルストーン（達成時期や区切り）
      └ グループ（任意の中間整理）
          └ 成果物（完成させるアウトプット）
              └ タスク（実際に作業する最小単位）
```

グループチケットは、マイルストーンと成果物の間に作成できる。必要に応じてグループを入れ子にし、マイルストーンと成果物の間を複数階層で整理できる。

### Project 以外

Ad-hoc / Explore など Project 以外の[管理区分](./work-management-categories.md)では、カテゴリ、グループ、成果物、タスクの階層で管理する。

```text
管理区分（Ad-hoc / Explore など）
  └ カテゴリ（仕事の種類）
      └ グループ（任意の中間整理）
          └ 成果物（完成させるアウトプット）
              └ タスク（実際に作業する最小単位）
```

Project 以外の[管理区分](./work-management-categories.md)でも、必要に応じてグループを入れ子にし、カテゴリと成果物の間を複数階層で整理できる。

タスクチケットは、作業者やAIが目的、作業内容、完了条件を理解し、見積もりとレビューができる粒度で作成する。1つのタスクチケットとして大きすぎる場合は、成果物やグループで整理し、その配下に実際に作業できる単位のタスクチケットを作成する。

## チケットに持たせる情報の考え方

チケットには、その階層で判断するために必要な情報を持たせる。具体的な項目は [チケット基本設計](../../basic-design/ticket-data-design.md) にまとめる。

### 上位チケット

Project、マイルストーン、カテゴリ、グループ、成果物チケットなどの上位チケットは、配下のチケットを整理し、目的や完了条件、目標状態を示すために使う。

上位チケットの進捗、ステータス、工数は、原則として配下のチケットから集約する。実際の作業はタスクチケットに分解して管理する。

### タスクチケット

タスクチケットは、実際に作業・レビューする単位として扱う。作業者やAIが、目的、作業内容、完了条件、関連情報を理解でき、見積もりとレビューができる粒度まで分解する。

タスクチケットには、担当者、現在ボールを持つ人、優先度、期限、見積工数、投下工数、関連リンク、ディスカッション、変更履歴などを紐づける。これにより、人間とAIが同じチケット情報を参照しながら作業を進められる。
