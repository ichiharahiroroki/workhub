# 一般的なチケット制との比較

このドキュメントでは、一般的なチケット制と WorkHub のチケット管理の違いを整理する。

## 一般的なチケット制

一般的なチケット制とは、仕事、依頼、課題、バグなどを「チケット」という単位で記録し、担当者、状態、優先度、期限、履歴を管理しながら、完了まで追跡する考え方である。

チケットは、対応漏れを防ぎ、誰が何を対応しているのか、どの状態にあるのか、どのような経緯で完了したのかを確認しやすくするために使う。

## WorkHub のチケット管理

WorkHub では、チケットを単なる作業管理の単位ではなく、仕事の文脈を集約する単位として扱う。

チケットには、何をするかだけでなく、目的、背景、判断履歴、関連する外部ツール、AI セッション、レビュー観点、見積工数、投下工数などを紐付ける。これにより、人間とAIが同じ文脈を参照しながら作業を進められる。

## 主な違い

| 観点 | 一般的なチケット制 | WorkHub のチケット管理 |
| --- | --- | --- |
| 主な目的 | 依頼、課題、バグ、作業を記録し、完了まで追跡する | 仕事の文脈をチケット単位で集約し、人間とAIが同じ情報を参照できるようにする |
| 管理する情報 | 担当者、状態、優先度、期限、履歴など | 目的、背景、判断履歴、外部ツール、AI セッション、レビュー観点、見積工数、投下工数など |
| 外部ツールとの関係 | チケット管理ツール内で情報を完結させることが多い | GitHub、Notion、Google Drive、Slack などを置き換えず、チケットに紐付ける |
| AIとの関係 | AI は補助機能として使われることが多い | 人間とAIが同じチケット情報を参照することを前提にする |
| 工数・リソース | 担当や作業時間を管理する場合がある | 見積工数、投下工数、進捗をもとに、実行可能性や計画の現実性を判断する |
| レビュー | コメントやステータスで確認する | チケット単位でレビュー対象とレビュー観点を整理し、このツール上でレビューできるようにする |
